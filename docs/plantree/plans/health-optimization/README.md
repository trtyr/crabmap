# Health Optimization Plan

## 目标

将 Crabmap 的健康评分从 13 提升到 80+，解决架构风险，提升代码质量。

## 范围

### 包含

1. 循环依赖重构
2. God modules 拆分
3. 死代码清理
4. 架构优化

### 不包含

1. 新功能开发
2. 性能优化
3. 文档完善

## 优先级

### P0（立即处理）

1. **循环依赖重构**
   - 重点：涉及 `web.rs`、`main.rs`、`ai.rs` 的循环
   - 方法：依赖反转、接口抽象
   - 预期：减少 50% 循环依赖

2. **analyzer.rs 拆分**
   - 问题：1217 行，92 个符号
   - 方法：按语法类型拆分为多个模块
   - 预期：每个模块 < 300 行

### P1（短期处理）

3. **query.rs 拆分**
   - 问题：836 行，41 个符号
   - 方法：按查询类型拆分
   - 预期：每个模块 < 200 行

4. **死代码清理**
   - 重点：高置信度死代码
   - 方法：移除或使用
   - 预期：清理 80% 死代码

### P2（中期处理）

5. **其他 god modules 拆分**
   - 包括：semantic.rs, ai.rs, web.rs, rag.rs
   - 方法：职责分离
   - 预期：每个模块 < 300 行

6. **架构优化**
   - 依赖方向调整
   - 接口抽象
   - 模块职责清晰化

## 执行计划

### 阶段 1：分析和准备（1-2 天）

1. 深入分析循环依赖
2. 设计重构方案
3. 准备测试用例

### 阶段 2：核心重构（3-5 天）

1. 重构 analyzer.rs
2. 重构 query.rs
3. 解决核心循环依赖

### 阶段 3：清理和优化（2-3 天）

1. 清理死代码
2. 优化模块结构
3. 更新测试

### 阶段 4：验证和收尾（1-2 天）

1. 运行完整测试
2. 验证健康评分
3. 更新文档

## 风险

### 高风险

1. **功能回归**：重构可能引入 bug
   - 缓解：充分测试，逐步重构

2. **性能下降**：模块拆分可能影响性能
   - 缓解：性能测试，优化关键路径

### 中风险

3. **时间超支**：重构可能比预期复杂
   - 缓解：分阶段执行，优先处理核心问题

4. **依赖冲突**：模块拆分可能引入新的依赖问题
   - 缓解：依赖分析，接口设计
